# Mutex ABI crash on Windows — findings from tracker_pi

A short write-up to share with the OpenCPN community. While bringing a new plugin (`tracker_pi`) to Windows we ran into a subtle runtime issue that seems worth flagging, since any future plugin using modern C++ synchronisation primitives is likely to hit the same thing. Everything below is offered in the spirit of "here's what we learned, maybe it helps someone" — not as criticism of OpenCPN or its packaging choices, which we very much appreciate.

---

## 1. What we observed

On the user's Windows navigation PC (OpenCPN 5.14), early beta builds of tracker_pi crashed the moment any mutex was touched:

- Every path that locked a `std::mutex` faulted with `EXCEPTION_ACCESS_VIOLATION 0xc0000005`, reading address `0x0`.
- Our SEH translator reported the fault inside MSVC STL's `_Mtx_lock`, on the `_Mutex_t` indirection pointer.
- Same instruction pointer each time. Every call site affected: scheduler configure, state-store update, toolbar refresh, dialog open, shutdown.
- macOS and Linux builds were unaffected. Development happens on macOS and Windows builds come out of CI (AppVeyor), so there was no local Windows environment to repro against — each diagnostic pass meant shipping a new beta and reading the crash output the user captured, which made the loop slow.

## 2. Root cause

Microsoft changed the binary layout of `std::mutex` in VS 2022 17.10 (microsoft/STL PR #4233, May 2024). Binaries built with the 17.10+ toolchain are ABI-incompatible with older `msvcp140.dll`.

OpenCPN 5.14 includes its own `msvcp140.dll` in the install directory, and Windows' DLL search order resolves that copy before the system-installed VC++ Redistributable. That's a perfectly reasonable choice for OpenCPN itself — it guarantees OpenCPN works on machines without an up-to-date redistributable. The side effect is that plugins compiled against post-17.10 STL headers end up running on the older runtime, and the `_Mutex_t` pointer is never initialised. The first `lock()` dereferences null.

This means the crash only shows up for a specific combination (plugin toolchain ≥ 17.10, target machine where OpenCPN's bundled runtime is older than the plugin's), so it wouldn't necessarily show up on every Windows install — just on the user's.

## 3. Why most existing plugins aren't affected

We took a look at a few well-established plugins to see how they handle synchronisation. Most of the older ones (e.g. `shipdriver_pi`) use **`wxMutex`** from wxWidgets, which is a thin wrapper over a Win32 `CRITICAL_SECTION`. It never goes through `msvcp140.dll`'s mutex machinery, so the ABI break doesn't apply.

Plugins that reach for the C++ standard library's synchronisation (`std::mutex`, `std::condition_variable`, `std::unique_lock<std::mutex>`, etc.) are the ones exposed — and only once they're built with a 17.10+ toolchain. So it's a quiet hazard for people writing new plugins in "modern" C++.

## 4. Options we considered

| # | Option | Notes |
|---|--------|-------|
| A | Wait for OpenCPN to ship a newer `msvcp140.dll`, or rely on the system redistributable. | Cleanest fix, but obviously not something a plugin author can drive on their own timeline. |
| B | Pin plugin builds to a pre-17.10 MSVC toolset (roll back the AppVeyor image). | Works today, but locks the toolchain indefinitely and turns every future dependency bump into a headache. |
| C | Avoid the affected STL primitives. Either `wxMutex` (matches what the rest of the plugin ecosystem does) or `std::shared_mutex` (backed directly by Win32 `SRWLOCK` in `kernel32.dll`, independent of `msvcp140.dll`). | Localised, survives future toolchain moves. |
| D | Drop to raw Win32 `SRWLOCK` / `CRITICAL_SECTION` under `#ifdef _WIN32`. | Works but adds platform-specific code for no real gain over C. |

## 5. What we ended up doing

We chose option **C**, specifically `std::shared_mutex`:

- Every `std::mutex` in the plugin became `std::shared_mutex` — `StateStore::mutex_`, `Scheduler::mutex_`, `OneTrackerPi::endpointStatusMutex_`. Call sites use `std::lock_guard<std::shared_mutex>` (exclusive lock). We never needed shared/reader locking, so semantics are identical to the original code.
- On MSVC, `std::shared_mutex` is implemented directly on top of `InitializeSRWLock` in `kernel32.dll`. Its layout is stable across the full MSVC 14.x ABI, which means the bundled `msvcp140.dll` is no longer in the picture.
- We also had to drop `std::condition_variable` — its MSVC implementation sits on top of `std::mutex` and inherits the same footprint. The scheduler's worker-thread wait became a 100 ms poll of `stopRequested_`. The tick cadence is 1 s, so the poll is invisible; shutdown latency rises from ~0 ms to at most 100 ms, which is fine for our use.
- We left long-form comments in the source documenting the ABI break, the interaction with the bundled runtime, and the conditions under which it would be safe to revert to `std::mutex` — so future maintainers don't "clean up" the workaround without understanding it.

Relevant commits in tracker_pi:
- `df50f8f` — switch all mutexes to `std::shared_mutex`, drop `condition_variable`.
- `3c9e7c8` — expand the source comments with the full 17.10 context.
- `f3744b4` — the SEH diagnostics (IP, access kind, faulting address) that narrowed this down in the first place.

## 6. Thoughts we'd like to share with the community

Offered as suggestions, in case they're useful to the core team or to other plugin authors:

1. **A note in the plugin author documentation** would likely save the next person some time — something along the lines of "if you're building on Windows with VS 2022 17.10 or later, prefer `wxMutex` or `std::shared_mutex` over `std::mutex` / `std::condition_variable` because of the 17.10 ABI change and the bundled runtime." We're happy to help draft this if it would be welcome.
2. **At some point down the road**, refreshing the bundled `msvcp140.dll` (or revisiting whether it needs to be bundled at all on modern Windows) would remove this sharp edge for everyone. We realise there are good reasons it's bundled today, and this is a long-term thought, not a request.

Thanks to everyone in the OpenCPN community for the support during the beta, and for all the work that makes the plugin ecosystem possible in the first place.
